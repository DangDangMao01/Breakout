## 1. Amazon Nova 2 模型家族

### 概述

AWS 发布了 4 个全新的 Nova 2 模型，覆盖从成本优化到多模态生成的完整场景。

### 1.1 Nova 2 Lite - 快速且经济的推理模型

**定位**: 日常工作负载的快速、经济型推理模型

**核心能力**:
- 输入: 文本、图像、视频
- 输出: 文本
- 上下文窗口: 100 万 token
- 支持微调 (SFT)

**性能基准**:
- 在成本效益方面优于同类模型
- 适合高频、低复杂度任务

**典型应用**:
- 客服聊天机器人
- 文档处理
- 内容审核
- 简单的自动化任务

**价格优势**: 最具成本效益的 Nova 模型

---

### 1.2 Nova 2 Pro - 最智能的推理模型

**定位**: AWS 最智能的推理模型，用于高度复杂的任务

**核心能力**:
- 输入: 文本、图像、视频、语音
- 输出: 文本
- 上下文窗口: 超大规模
- 内置 Web 搜索和代码执行

**性能基准**:
- 在复杂推理任务中表现卓越
- 代码生成能力强
- 工具使用准确率高

**典型应用**:
- 复杂代码生成
- 高级数据分析
- 多步骤推理任务
- AI Agent 开发

**独特功能**:
- **内置 Web Grounding**: 自动搜索最新信息
- **代码执行**: 可以运行 Python 代码进行计算

---

### 1.3 Nova 2 Sonic - 语音对话 AI

**定位**: 语音到语音的对话 AI 模型

**核心能力**:
- 统一文本和语音理解与生成
- 实时、人性化的对话体验
- 支持多语言和多种口音
- 自然的语速和语调

**集成能力**:
- 与 Amazon Connect 深度集成
- 支持第三方 ASR/TTS（Deepgram、ElevenLabs）

**典型应用**:
- 智能客服系统
- 语音助手
- 实时翻译
- 电话自动化

**性能特点**:
- 低延迟响应
- 自然的对话流
- 上下文理解能力强

---

### 1.4 Nova 2 Omni - 统一多模态生成模型 ⭐ 行业首创

**定位**: 行业首个统一的多模态推理和生成模型

**核心能力**:
- 输入: 文本、图像、视频、语音
- 输出: **文本 + 图像**（同时生成）
- 上下文窗口: 75 万词 + 数小时音频 + 长视频 + 百页文档
- 同时分析: 整个产品目录、客户评价、品牌指南、视频库

**革命性突破**:
- **统一处理**: 同时处理所有模态
- **双重生成**: 在单一工作流中生成文本和图像
- **降低成本**: 无需连接多个专用模型
- **简化复杂度**: 一个模型完成多模态任务

**实际案例**:
营销团队可以同时分析产品详情、客户评价、品牌指南和视频库，然后立即生成完整的营销活动，包括标题、文案、社交媒体帖子和视觉内容——全部在单一工作流中完成。

**典型应用**:
- 营销内容生成（文案 + 图片）
- 产品设计（描述 + 原型图）
- 教育内容创作
- 多媒体报告生成

### 1.5 Nova Forge - 构建自己的前沿模型 🔥 行业首创

**定位**: 首个"开放训练"服务，让企业构建自己的定制模型

**解决的问题**:

当前企业在集成专有知识到 AI 时面临三个困境：
1. **表面定制**: 定制专有模型只能触及表面
2. **回归风险**: 继续训练开源模型可能丢失基础能力
3. **高成本**: 从头构建需要巨额投资

**Nova Forge 的解决方案**:

提供对 Nova 模型的**预训练、中期训练、后期训练检查点**的独家访问，让客户在每个训练阶段混合自己的专有数据和 Amazon Nova 策划的数据集。

**核心能力**:
- 访问 Nova 的完整训练检查点
- 混合专有数据和 Nova 数据集
- 在每个训练阶段进行定制
- 保留 Nova 的基础能力

**结果**:
一个结合了 Nova 完整知识和推理能力，同时深度理解企业特定业务需求的定制模型。

**客户案例**:

**Reddit**:
> "使用 Nova Forge 让我们能够用更统一的系统改进 Reddit 的内容审核，已经取得了令人印象深刻的结果。我们正在用一个更准确的解决方案替换多个不同的模型，使审核更高效。用一个统一的方法替换多个专用 ML 工作流，标志着我们在 Reddit 上实施和扩展 AI 的方式发生了转变。"

**其他早期客户**:

**定价**:
- 预计成本: 约 10 万美元起

**部署**:
定制模型可以直接部署到 Amazon Bedrock，享受企业级安全、可扩展性和数据隐私。

## 2. Amazon Bedrock AgentCore

### 概述

Amazon Bedrock AgentCore 是 AWS 为 AI Agent 提供的完全托管基础设施服务，解决了 AI Agent 从原型到生产的核心挑战。

### 2.1 核心问题

**开发者面临的挑战**:
- 用 LangChain 和 LLM API 构建的酷炫原型大多停留在试点和演示阶段
- AI 难以扩展
- 生产环境风险高
- 需要大量胶水代码
- 缺乏真正的安全性和可观测性

**AWS 的解决方案**:
AgentCore 是 AWS 对"客户如何构建可靠扩展的自己的 Agent"的答案。

---

### 2.2 AgentCore 核心服务

#### 2.2.1 AgentCore Runtime

**功能**:
- 沙箱化的低延迟无服务器环境
- 会话隔离
- 支持任何 Agent 框架（CrewAI、LangGraph、LlamaIndex、Google ADK、OpenAI Agents SDK、Strands Agents）
- 处理多模态工作负载
- 支持长时间运行的 Agent

**价值**:
开发者无需管理基础设施，专注于构建更智能的 Agent。

---

#### 2.2.2 AgentCore Policy ⭐ re:Invent 2025 新增

**功能**:
使用自然语言设置 Agent 边界，控制 Agent 可以使用哪些工具以及允许执行哪些操作。

**工作原理**:
- 用自然语言编写策略
- AgentCore 自动转换为 Cedar（AWS 开源授权策略语言）
- AgentCore Gateway 大规模执行这些规则
- 测试显示每秒处理数千个请求，策略检查仅需毫秒

**实际案例**:
可以执行如"Agent 只能在工作时间访问客户数据"的规则，无需编写任何自定义授权逻辑。

**价值**:
解决了 Agent 生产部署的最大障碍之一——对不受限制的工具访问的担忧。

---

#### 2.2.3 AgentCore Policy Evaluation ⭐ re:Invent 2025 新增

**功能**:
13 个内置评估器，持续评估 Agent 行为的正确性、有用性、安全性和工具选择准确性。

**核心能力**:
- 实时质量评分
- CloudWatch 集成
- 支持自定义评估器
- 情节记忆（Episodic Memory）跟踪推理模式
- 帮助 Agent 随时间改进

**价值**:
- 移除了创建内部评估或评分框架的负担
- 默认提供内置质量监控和 CloudWatch 警报
- 情节记忆让 Agent 自动从先前交互中学习

---

#### 2.2.4 AgentCore Episodic Memory 🧠 re:Invent 2025 正式发布

**定位**: Agent 真正智能的核心——记忆和上下文

**什么是情节记忆？**

情节记忆允许 Agent 保留和引用过去的交互，使其能够以类似人类的方式回忆和推理先前的经验。

**AI 记忆类型**:

| 记忆类型 | 说明 | 示例 |
|---------|------|------|
| 短期记忆 | 工作上下文 - 即时输入、当前任务状态 | 当前对话内容 |
| 长期记忆 | 随时间积累的知识 - 模式、经验、外部数据 | 训练数据 |
| 语义记忆 | AI 的"知识库" - 事实、概念、关系 | "巴黎是法国首都" |
| 程序记忆 | AI 的"操作知识" - 执行步骤序列 | 如何排序列表 |
| **情节记忆** | **特定经验或事件 - Agent 的个人历史** | **"用户上次旅行时喜欢靠窗座位"** |

**与之前的区别**:

之前的 AgentCore 记忆模型需要管理独立的短期和长期记忆存储，而情节记忆让 Agent 可以在特定条件触发时自动回忆相关信息，减少或消除了自定义指令的需求。

**工作原理**:

1. **捕获**: 捕获用户和系统交互的有意义片段
2. **压缩**: 识别重要时刻，压缩成紧凑摘要
3. **组织**: 组织以便高效、无噪音的检索
4. **反思**: 回顾过去时刻，发现模式和有意义的教训

**实际案例**:

David Richardson (VP AgentCore):
> "这个想法是帮助捕获用户真正希望 Agent 记住的信息，例如，'他们在家庭旅行时喜欢的飞机座位是什么？'或'他们寻找的价格范围是多少？'"

**客户案例**:

**S&P Global Market Intelligence**:
> "随着数百个专用 Agent 的出现，管理状态和维护一致的上下文变得越来越困难，突显了对统一记忆层的需求。Amazon Bedrock AgentCore Memory 通过我们的多 Agent 编排堆栈提供无缝、集中的状态检查点，提供了解决方案。借助新的情节记忆功能，我们的 Agent 将从先前的分析中学习，以生成更智能的洞察。"
> — Helene Astier, 技术主管, S&P Global Market Intelligence

**典型应用**:
- 客户支持和服务（记住用户的先前问题和解决方案）
- 个性化（存储用户偏好和长期上下文）
- 故障排除和诊断（访问过去的"情节"以识别模式）

**价值**:
情节记忆本质上是 Agent 选择和创建自己的知识库。可以把它想象成 Agent 的日记。

## 4. Amazon S3 Vectors

### 概述

**定位**: 首个具有原生向量支持的云对象存储

**发布状态**: 正式发布（GA）

**核心突破**: 将向量索引和相似性搜索直接集成到 S3 中，无需单独的专用数据库。

---

### 4.1 核心能力

**规模**:
- 单个索引: 最多 **20 亿个向量**
- 每个存储桶: 最多 **20 万亿个向量**（跨 10,000 个索引）

**性能**:
- 频繁查询: 平均 **100 毫秒**
- 不频繁查询: **< 1 秒**
- 每次返回最多 **100 个结果**

**集成**:
- Amazon Bedrock Knowledge Bases
- Amazon OpenSearch Service
- 支持混合搜索（关键词 + 向量）

---

### 4.2 性能数据（预览期真实数据）

| 客户 | 向量数量 | 查询性能 | 应用场景 |
|------|---------|---------|---------|
| BMW | 数十亿 | 亚秒级 | 推荐系统 |
| MIXI | 数亿 | < 1 秒 | 语义搜索 |
| Qlik | 数亿 | 亚秒级 | 数据分析 |
| Twilio | 数亿 | < 1 秒 | 客户支持工作流 |

---

### 4.3 成本优势

**AWS 声称**: 相比大多数专用向量数据库节省 **90% 成本**

**定价示例**:

| 存储向量数 | 每月查询数 | S3 Vectors 月成本 |
|-----------|-----------|------------------|
| 1 亿 | 100 万 | $11.38 |
| 4 亿 | 1000 万 | $1,217 |

**对比**: Pinecone、Weaviate 等竞品在类似规模下的成本通常高得多，特别是查询流量增加时。

---

### 4.4 使用场景

**最适合**:
- 后台 RAG 管道
- 批量语义搜索
- 推荐引擎
- 文档理解
- 不需要超低延迟（< 10ms）的场景

**不适合**:
- 需要亚 10 毫秒延迟的场景
- 极高 QPS（每秒查询数）的场景

**价值定位**: 补充而非完全替代专用向量数据库

---

### 4.5 使用示例

```bash
# 创建向量索引
aws s3vectors create-index --bucket my-vector-bucket --name product-embeddings

# 查询向量
aws s3vectors query --index product-embeddings --vector file://query.json
```

### Keynote 3: Dr. Werner Vogels (Amazon CTO) - "文艺复兴开发者" 🎭 最后一次演讲

**历史意义**: Werner Vogels 在担任 14 年后的最后一次 re:Invent 主题演讲

**核心观点**:
> "我们再次处于文艺复兴时代，你们就是新的文艺复兴开发者。创造力和技术共同进化。好奇心引领学习和发明。"

**关键问题**: "AI 会取代我的工作吗？"

**Werner 的回答**: 
> "也许会。AI 会让我过时吗？绝对不会……如果你进化的话。"

**文艺复兴开发者的五大原则**:

#### 1. 保持好奇心，持续学习 (Be Curious and Keep Learning)
- **核心**: 好奇心对于持续学习和发明至关重要
- **实验**: 包括愿意失败，这对于学习系统行为至关重要
- **Werner 的经历**: 从学校学习 68000 汇编和 COBOL，到现代 IDE 如 Cursor 和 Kiro
- **关键**: 适应新工具，不断学习

#### 2. 系统性思考 (Think in Systems)
- **定义**: 理解系统的互联部分如何相互影响
- **重要性**: 构建弹性系统的关键
- **要求**: 超越孤立组件的思考
- **目标**: 理解整体架构和依赖关系

#### 3. 精确沟通 (Communicate with Precision)
- **核心**: 清晰的沟通，特别是通过规范
- **AI 时代的重要性**: 减少 AI 辅助编码中的歧义
- **目标**: 确保机器创建的逻辑与人类意图一致
- **挑战**: AI 生成代码的速度超过开发者理解的速度

#### 4. 成为所有者 (Be an Owner)
- **责任**: 即使使用 AI，开发者也必须对代码质量负责
- **验证深度**: AI 可以比开发者理解更快地生成代码，创造"验证深度"风险
- **质量保证**: 确保代码符合监管要求，避免幻觉
- **机制 vs 良好意愿**: 良好意愿不足以确保质量，需要具体机制
- **关键引用**: "工作是你的，不是工具的。你构建它，你拥有它。"

#### 5. 成为博学者 (Become a Polymath)
- **T 型开发者**: 在一个领域拥有深厚专业知识，同时对其他技术和学科有广泛理解
- **广度 + 深度**: 既要专精，也要博学
- **适应性**: 能够跨领域工作和学习

**隐藏的工作之美**:
> "开发者应该为那些看不见的系统、干净的部署和回滚感到自豪，这些确保了卓越运营，即使客户可能永远不会注意到它们。"

**总结**:
Werner 的告别演讲不仅是对开发者的职业建议，更是一本适用于 2026 年所有人的职业手册。因为每份工作都在成为技术工作。
